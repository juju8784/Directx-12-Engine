107  //! Launch a multi-thread spanning simulatenous array modification operation to run concurrently with this thread.
108  /*!
109  * Allows you to process large contigious data sets using a divide & conquer approach.
110  * Based on how many "_maxSections" are required to traverse "_arraySize" we create that many jobs in the internal thread pool.
111  * Then each job will run "_parallelTask" for each input/output pair in it's designated section.(optional array location is provided)
112  * To know when a single section completes listen for: PARALLEL_SECTION_COMPLETE, There is no cross section ordering garuntee.
113  * Global resources accessed by the provided operation are your responsibility to synchronize for thread safety. (recc: GThreadShared)
114  * However the provided arrays are safely accessed due to being divided evenly amongst the internal jobs. (no overlapp)
115  * You may wait for this entire operation to complete using "Converge" or listen for: PARALLEL_TASK_COMPLETE
116  * Once you have "Converged" or received the appropriate message(s) you are safe to acd amongst processing cores.
117  *
118  * \param [in] _parallelTask Pointer to static operation responsible for element processing, called once for each specific element.
119  * \param [in] _maxSection The maximum # of elements processed by a thread at once, this has the largest impact on perfomance.
120  * NOTE: As rule of thumb this value should be smaller for complex operations and larger for simplistic ones.
121  * \param [in] _arraySize The size of array or problem domain that is to be divideey wish each routine to have access to.
122  * \param [in] _userData `(optional)` Pointer to some custom user provided data that the input/output data arrays might need.
123  * \param [in] _inStride `(optional)` if 0 then sizeof(Input) type used for input array traversal, otherwise this byte amount is used.
124  * \param [in] _inputArray `(optional)` Array of input data elements that will be fed to the processing system.
125  * \param [in] _outStride `(optional)` if 0 then sizeof(Output) type used for output array traversal, otherwise this byte amount is used.
126  * \param [out] _outputArray `(optional)` Array of output data elements that will be fed to the processing system.
127  *
128  * \retval GReturn::MEMORY_CORRUPTION _inputArray & _outputArray have an overlapping memory space. (Just use _outputArray instead)
129  * \retval GReturn::INVALID_ARGUMENT You must provide a non-null function to execute and a non-zero arraySize & maxSection.
130  * Cannot use the same array for _inputArray and _outputArray. In this situation, just use _outputArray.
131  * \retval GReturn::SUCCESS We have successfully submitted the task for processing.
132  */
133  GATEWARE_TEMPLATE_FUNCTION(BranchParallel)
134  //! Forces the current thread to wait until any & all branched processing has completed.
135  /*!
136  * Polls the GConcurrent until all running tasks have completed.
137  * Use this when you need to access resources on the current thread that may still be processing elsewhere.
138  * If you want a less invasive method of waiting and responding, consider using the built-in messaging system to be notified instead.
139  *
140  * \param [in] _spinUntil Determine how long the current thread will spinlock waiting for all Tasks to complete. (in NANOSECONDS)
141  * Once "_spinUntil" nanoseconds has elapsed the current thread will yield to the OS scheduler. (1 millisecond yield)
142  * Upon being re-scheduled the thread will try to spinlock again until "_spinUntil" is reached again or completion.
143  * NOTE: If your not in a time/performance critical area you should just use 0. Wake times dependent on OS resolution.
144  *
145  * \retval GReturn::FAILURE There have been no Tasks ever or since you last Converged.
146  * \retval GReturn::SUCCESS Tasks were running and we sucessfully waited for their completion.
147  */
148  GATEWARE_FUNCTION(Converge)
